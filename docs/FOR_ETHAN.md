## 我们具体干了啥(分三层,客观说)

### 1) 数据库表层
- **删掉了树里没有的东西**:整套「地图/打卡/足迹/周排行」(user_locations、landmarks、explorations…)和整套「朋友圈式社交动态」(posts、comments、likes…)。这些表、策略、函数、Realtime 订阅全清了。
- **profiles(用户表)大改**:
  - 短号从 `sudo_id` 改名 **`zzup_id`**(app 改名 zZuP! 了)。
  - 加了 `gender`、`onboarded`、`deleted_at`(软删)、宠物阶段 `pet_stage`(幼/青/成)、漂流瓶配额(按等级 1/3/5)。
  - 隐私加了:真名是否可被搜、加好友途径开关、5 类通知开关。
  - 删了旧的地理/排行字段。
  - 敏感信息(邮箱、生日)**在数据库层就锁死**,别人怎么查都查不到(生日只给「年龄」)。
- **好友系统重做**:三种状态 pending/rejected/accepted;同一时刻两人之间只能有一条待处理请求;被拒绝后可以重新发(新的一条)。
- **拉黑改成「按身份」**:你能单独拉黑某人的**宠物马甲**而不影响和他真人的好友关系(他也不会知道被拉黑的是宠物)。
- **聊天彻底重构成「四窗口」模型**(最大的一块):
  - 一张 `conversations` 表装下所有会话类型:固定宠物会话、群聊、私聊、(以及将来的 Pet Chat / 漂流瓶)。
  - 两个用户之间**最多 4 个独立私聊窗口**:双真人 / 双宠物 / 一人一宠 ×2,后端保证**绝不合并、绝不暴露这几个窗口是同一个人**。
  - 每条消息带「这条是真人发的还是宠物发的」标记,前端据此换头像。

### 2) RPC 层(数据库里的"安全函数")
- 几乎所有**写操作**(加好友、拉黑、建群、开私聊…)都封装成数据库函数,前端只能调函数、不能乱改表 —— 防作弊、防绕过权限。
- 补了一批**读取函数**:搜索用户、好友列表、待处理请求、关系状态、会话列表(都自动按对方的展示身份过滤、自动隐藏过期窗口)。

### 3) lib/api(前端数据层)
- `auth / friends / conversations / messages / _xp` 全部重写,对齐新表新函数,TypeScript 编译干净。
- 旧的 `groups.ts`(指向已删的旧表)删了,换成 `conversations.ts`。

## 几个你必须知道的设计点(大白话)

- **宠物不是独立账号**,就是你账号的另一个"身份面"。一人一只宠物。
- **加好友只在真人之间**。宠物会话里想加好友:宠物方在对话框**报出自己的 zZuP id**,对方按真人加。(唯一例外:Pet Chat 走"≥3 条消息→加好友按钮"那套,加完这个窗口**原地变成**真人好友会话,宠物 AI 的历史还能往上翻 —— 这部分逻辑后面建。)
- **S_A** = 用户给别人看自己时的形态:只给看真人 / 只给看宠物 / 同框都给看。
- **匿名(重要提醒)**:这一版匿名是"软"的 —— 接口里其实还能拿到对方真实账号 id。前端**先别把它当成铁桶匿名**,真正的强匿名我们后面再升级。
