## ✅ 问题 #1：用户可自改 edu_verified / pet_xp / pet_level / sudo_id 等受保护列

**严重级别：** 🔴 CRITICAL
**完成时间：** 2026-04-17

### 漏洞描述

`profiles` 表上唯一的 UPDATE policy 只限制"只能改自己这行"（`auth.uid() = id`），不限制**哪些列**可改。恶意客户端可以：

```typescript
await supabase.from('profiles').update({
  edu_verified: true,
  university: 'Stanford',
  pet_xp: 99999,
  pet_level: 1000,
}).eq('id', myUserId)
```

完全绕过 Offer 验证和 XP 系统。

### 云端验证过程

- Query 1（policies）：确认只有一条 UPDATE policy 且无 `with_check`
- Query 2（triggers）：profiles 上无任何 trigger
- Query 3（column privileges）：authenticated 对所有 24 列都有 UPDATE（实际 28 列，其中 4 列因为 migration 54 未推被遗漏）
- **顺带发现**：`54_profile_visibility.sql` 从未在云端执行过（migration 脱漏）

### 修复措施

1. **先补跑 migration 54**（加 `profile_visibility`、`show_date_of_birth`、`show_nationality`、`show_qr_code` 4 个列）
2. **新增 migration 55**（[55_protect_profile_columns.sql](../supabase/migrations/55_protect_profile_columns.sql)）：
   - `REVOKE UPDATE ON public.profiles FROM authenticated / anon`
   - `GRANT UPDATE (21 个白名单列) ... TO authenticated`
3. **[auth.ts](../lib/api/auth.ts)** 的 `ProfileUpdate` 类型注释更新，解释保护列分工

### 保护列最终清单

**用户可改（21 列）：**
real_name, bio, avatar_url, qr_code_url, date_of_birth, nationality, region, university, personal_email, edu_email, pet_name, pet_avatar_url, pet_bio, identity_mode, location_sharing, ranking_opt_in, ranking_identity_mode, profile_visibility, show_date_of_birth, show_nationality, show_qr_code

**受保护（7 列）：**
| 列 | 写入路径 |
|---|---|
| `edu_verified` | verify-offer Edge Function / 未来邮箱验证 |
| `personal_email_verified` | 未来邮箱验证流程 |
| `pet_xp`, `pet_level` | 只能走 `add_xp()` RPC |
| `sudo_id` | 系统分配，注册时自动生成 |
| `id`, `created_at` | 不可变 |

### 验证结果

云端重跑 Query 3，返回的 21 列与 GRANT 列表完全匹配，保护列不在其中。✅

### 遗留事项（新增 TD-13）

**TD-13**：用户修改 `university` 时应自动重置 `edu_verified=false`。

**原因**：如果用户 Offer 验证通过（`edu_verified=true, university='MIT'`），之后在 app 里把 university 改成 'Harvard'，当前不会重置 edu_verified，会被错误归入 Harvard 的排行榜和 edu_verified 群。

**延后原因**：需要 `BEFORE UPDATE` trigger，和 Edge Function 的"一次写两列"逻辑强耦合。Module 7 做 Offer 验证时一起加。

**修复时机**：Module 7 开工时

## ✅ 问题 #2：profiles SELECT 全字段公开

**严重级别：** 🔴 CRITICAL
**完成时间：** 2026-04-17

### 漏洞描述

`profiles` 表的 SELECT policy 只要求 `auth.uid() IS NOT NULL`（登录即可），无任何列级限制。任何登录用户可 `.from('profiles').select('*').eq('id', any_user_id)` 读到全站用户的 `personal_email`、`edu_email`、`date_of_birth`、`nationality`、`region`、`qr_code_url`、`show_*` 隐私开关、`location_sharing` 等 PII。

[auth.ts getProfile](../lib/api/auth.ts) 的 JS 层隐私过滤（旧 TD-4/TD-5 修复）只是装饰 —— 绕开 getProfile 直接查表就能拿到全部数据。

### 修复措施

**方案：服务端过滤 + 列级 SELECT 保护（Option B 深修）**

1. **Migration 25 重写**（合并 54、55）：
   - REVOKE ALL ON profiles FROM authenticated/anon
   - GRANT SELECT 只给 15 个可在 JOIN 中使用的安全列（姓名、头像、大学等）
   - REVOKE 掉的敏感列：`personal_email`, `edu_email`, `personal_email_verified`, `date_of_birth`, `nationality`, `region`, `qr_code_url`, `show_*` 3 个开关, `location_sharing`, `ranking_opt_in`, `ranking_identity_mode`
   - 新增 RPC `get_my_profile()` (SECURITY DEFINER) —— 本人读自己完整资料
   - 新增 RPC `get_other_profile(target_id)` (SECURITY DEFINER) —— 服务端按对方隐私设置做过滤

2. **auth.ts 的 `getProfile()` 重写**：
   - `select *` → `rpc('get_my_profile')` 或 `rpc('get_other_profile', { target_id })`
   - JS 层隐私过滤代码全部删除（80 行 → 30 行）
   - 过滤逻辑搬到 DB，无法绕过

### 账号软删除 / 内容保留哲学（2026-04-17 确认）

**SUDO 的删除模型：**
- 用户"注销账号" = **软删除**（auth.users + profiles 行保留，标记为已注销状态）
- 该用户发的**帖子、评论、消息**全部保留，UI 显示为"已注销用户"
- 别人和该用户的**对话、互动**保留

**为什么这么设计：**
- 防止恶意用户发完违规内容立刻注销逃避追责
- 保留内容供事后审查 / 投诉处理 / 法律取证

**对 FK 设计的影响：**
- 内容类表（`messages.user_id`、`posts.user_id`、`comments.user_id`）→ `ON DELETE SET NULL`，内容保留作者显示为"已注销"
- 关系类表（`friendships`）→ `ON DELETE SET NULL`，记录留存作为审计痕迹
- 操作类表（`blocked_users`、`group_members`）→ `ON DELETE CASCADE`，无审计价值
- 系统类表（`user_locations`、`explored_paths`）→ `ON DELETE CASCADE`，无意义记录

**注：** 这个设计意味着 profiles 在正常流程下**几乎不会被真正删除**。FK 的 ON DELETE 行为只在罕见的"硬删除"场景生效（管理员清理 / 用户主动彻底删除）。

### blocked_users 透明可见（2026-04-17 确认）

**当前行为：** 被拉黑的用户能查到"谁拉黑了我"。

### sudo_id 顺序递增（原 Code Review 问题 #4）

**现状**：`sudo_id` 由 `sudo_id_seq` 顺序生成，5 位数字从 `00001` 开始。

**Code Review 曾标记为问题**（枚举攻击风险），但**这是刻意的产品设计**：
- 稀缺性体现 —— 第一个注册的用户 ID 必然是 `00001`
- 低号码 = 早期用户身份象征
- 类似 Twitter 早期短用户名、QQ 靓号的价值感

**产品权衡已接受**：
- 风险：攻击者可按顺序扫描 00001-99999 拿到所有已注册用户（但配合问题 #2 修复后，能拿到的信息会很有限）
- 收益：早期用户荣誉感 / 品牌记忆点

**保持原设计，不修复。** 但问题 #2 修好后攻击者即使枚举出 sudo_id 列表，拿不到敏感 PII，风险可控。
